Promote new version to Production environment

Step 1 - Deploy the new version

We will start deploying a new version v1.1.1 in the offline color. And get it ready to receive online user’s traffic

These are the main tasks that we have to execute:

  • Set the new image tag values in the offline color.

  • Change the application configuration values to use the online services.

  • Scale up the offline color.

  • Commit the changes in a new branch and create a pull request.

If we do not automate those tasks, we will have to see which is the offline color and make all the changes in the right values files. However, we can automate it in a pipeline. The pipelines will find the offline color, make the changes in the right values files and create the pull request.

  • Manual steps

  • Automatic pipeline

We can get the offline color from Openshift service or from the file values/products/envs/prod/non-static-networking-values.yaml. Then edit the following files in the right environment and color.

cd helm/quarkus-helm-umbrella/chart

  • Set new tag image values in the offline color.

Edit the file values/products/envs/prod/green/image-tag-values.yaml. Set its content to deploy new version v1.1.1

products-green:
  quarkus-base:
    image:
      tag: v1.1.1

  • Change the application configuration values to use the online services.

  • Scale Up the offline color.

Edit the file values/products/envs/prod/green/non-static-values.yaml. Set this content to use the online backend and scale up.

products-green:
  quarkus-base:
    replicaCount: 2
  mode: online

Commit the changes in a new branch and push them to your repository:

git checkout -b new-image-v1.1.1
git add values/products/envs/prod/green
git commit -m 'products-green  new image tag v1.1.1'
git push origin new-image-v1.1.1

Now, create a pull request from the new-image-v1.1.1 to the release branch:

Figure 1. Pull request notification on GitHub

We have created a pipeline that:

  • Gets the offline color in the designed environment.

  • Changes the right files to deploy a new version.

  • Commits the changes in a new branch and creates a pull request.

apiVersion: tekton.dev/v1beta1
kind: PipelineRun
metadata:
  generateName: shop-products-new-version-
  namespace: %USER%-continuous-deployment
spec:
  pipelineRef:
    name: pipeline-blue-green-environments    
  params:
  - name: APP
    value: products
  - name: NEW_IMAGE_TAG
    value: "v1.1.1"
  - name: STEP
    value: "new-version"
  - name: REPLICA_COUNT
    value: "2"   
  - name: NAMESPACE
    value: "%USER%-prod"
  - name: REPO_FULL_NAME
    value: %REPOSITORY%/cloud-native-deployment-strategies
  - name: ENV
    value: prod
  workspaces:
  - name: app-source
    persistentVolumeClaim:
      claimName: workspace-pvc-shop-cd-new-version
oc create -f 1-pipelinerun-products-new-version.yaml

You should have the following pull request:

ArgoCD will synchronize the application with the new changes. You can do it manually but to make it faster, click on the SYNC button.

Step 2 - Switch new version to Online

We are going to open the new version to final users. To do so, we just have to change the online service to use the other color that we have ready.

These are the main tasks that we have to execute:

  • Change the online service.

  • Commit the changes in a new branch and create a pull request.

  • Manual steps

  • Automatic pipeline

cd helm/quarkus-helm-umbrella/chart

  • Change the online service

Edit the file values/products/envs/prod/non-static-networking-values.yaml. Set this content to use the green color in the online service.

productsNetworkingOnline:
  version: green
productsNetworkingOffline:
  version: green

Commit the changes in a new branch and create a pull request

git checkout -b switch-to-colout-green-image-v1.1.1
git add values/products/envs/prod/non-static-networking-values.yaml
git commit -m 'products switch online to green'
git push origin switch-to-colout-green-image-v1.1.1

Create a Pull Request on the GitHub website like in the previous step.

We have created a pipeline that:

  • Get the offline color in the designed environment.

  • Execute an E2E test to validate the version.

  • Change the right files for the online service.

  • Commit the changes in a new branch and create a pull request.

apiVersion: tekton.dev/v1beta1
kind: PipelineRun
metadata:
  generateName: shop-products-switch-
  namespace: %USER%-continuous-deployment
spec:
  pipelineRef:
    name: pipeline-blue-green-environments   
  params:
  - name: APP
    value: products
  - name: STEP
    value: "switch"
  - name: NEW_IMAGE_TAG
    value: "v1.1.1"
  - name: NAMESPACE
    value: "%USER%-prod"
  - name: REPO_FULL_NAME
    value: %REPOSITORY%/cloud-native-deployment-strategies
  - name: ENV
    value: prod
  workspaces:
  - name: app-source
    persistentVolumeClaim:
      claimName: workspace-pvc-shop-cd-new-version
oc create -f 2-pipelinerun-products-switch.yaml

Step 2,5 - Rollback

Imagine that something goes wrong, we know that this never happens but just in case. We can do a very quick rollback just by undoing the change in the Products online service. But, are we sure that with all the pressure that we will have at this moment, we will find the right service and change the label to the right color? Let’s move this pressure to the pipeline.

Those are the main tasks that we have to execute:

  • Change the online service

  • Commit the changes in a new branch and create a pull request.

  • Automatic pipeline

Step 3 - Align and scale down Offline

Finally, when online is stable we should align offline with the new version and scale it down. It does not make sense to use the same resources that we have in the online deployment.

These are the main tasks that we have to execute:

  • Set new tag image values in the new offline color.

  • Change the application configuration values to use the offline services.

  • Scale down the new offline color.

  • Change the offline service to use the new offline color.

  • Commit the changes in a new branch and create a pull request.

  • Manual steps

  • Automatic pipeline

We can get the new offline color from Openshift service or from the file values/products/envs/prod/non-static-networking-values.yaml. Then edit the following files in the right environment and color.

cd helm/quarkus-helm-umbrella/chart

  • Set new tag image values in the new offline color. Edit the file values/products/envs/prod/blue/image-tag-values.yaml. Set this content to deploy new version v1.1.1

products-blue:
  quarkus-base:
    image:
      tag: v1.1.1

  • Change the application configuration values to use the offline services.

  • Scale Down the new offline color. Edit the file values/products/envs/prod/blue/non-static-values.yaml. Set this content to use online backend and scale up.

products-blue:
  quarkus-base:
    replicaCount: 1
  mode: offline

  • Change the offline service to use the new offline color. Edit the file values/products/envs/prod/non-static-networking-values.yaml. Set this content to use the blue color in the offline service.

productsNetworkingOnline:
  version: green
productsNetworkingOffline:
  version: blue

Commit the changes in a new branch and create a pull request

git checkout -b align-offline-v1.1.1
git add values/products/envs/prod/blue
git add values/products/envs/prod/non-static-networking-values.yaml
git commit -m 'products-blue env prod align image tag v1.1.1'
git push origin align-offline-v1.1.1

We have created a pipeline that:

  • Get the offline color in the designed environment.

  • Change the right files to align the color and scale down.

  • Commit the changes in a new branch and create a pull request.
